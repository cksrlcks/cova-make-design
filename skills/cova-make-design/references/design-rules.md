# 디자인 규칙 (사람이 만든 시안처럼)

## 0. 생성 프로세스 — 2-패스 (건너뛰기 금지)

**유형 분기 먼저:** 메인·서브 페이지는 이 파일 + [design-trends.md](design-trends.md)(방향 카탈로그),
관리자·도구형 제품은 이 파일의 §2~§4 공통 규칙 + [saas-admin.md](saas-admin.md)를 따른다.

**1차 패스 — 디자인 플랜 선언.** 코드를 쓰기 전에 다음을 확정하고, **완성 HTML 최상단에
design-plan 주석으로 기록**한다(주석이므로 화면에는 노출되지 않는다 — 자체 테스트가 이 주석과
실제 CSS의 일치를 대조한다):

```html
<!-- design-plan: 후보 ①…(기각 사유) ②… ③… | 커밋: D3 다크 시네마틱 | 팔레트: #0b0c0f/#f2f2f0/#e8a33d…
     | 타이포: Paperlogy 800 × Pretendard | 시그니처: … | 볼륨: 균형(무브 2·6·19) | 비튼 축(D2·D4만): …
     | 섹션맵: hero / 좌텍우이미지 / 풀블리드미디어 / 리스트단계 / 괘선표 / 하단CTA
     | 라이브: 채택[…] 회피[보라 그라데이션·마키] (웹 미가용 시 "미조회(문서 기준)") -->
```

- **방향**: design-trends.md 카탈로그에서 후보 3개 나열 → 관행 기각 → 1개 커밋 (어드민이면 무드 A/B/C).
- **팔레트**: 이름 붙인 hex 4~6개 (예: 지면 #f7f5f0 / 잉크 #191817 / 딥그린 #1f3d2b / 브라스 #9a7b4f).
- **타이포**: 헤드라인 폰트 × 본문 폰트 × (선택) 모노 — 역할과 웨이트까지.
- **레이아웃 콘셉트**: 한 문장 (예: "좌측 고정 목차 + 우측 스크롤 아카이브, 섹션마다 폭이 다른 편집 구조").
- **시그니처 요소 1개**: 이 페이지가 기억될 단 하나.
- **섹션 구성**: 브리프에서 도출한 순서 (고정 스켈레톤 금지 — 아래 §2).
- **섹션맵**: 각 섹션의 레이아웃 패밀리를 코드 전에 선언한다(hero / 좌텍우이미지 / 우텍좌이미지 /
  풀블리드미디어 / 넓은이미지+짧은설명 / 리스트·단계 / 가로스크롤 / 괘선표 / 하단CTA …).
  **좌우만 반전한 것은 같은 패밀리로 본다.** §2의 다양성·상한을 이 선언으로 강제한다.
- **라이브 신호(3.5단계)**: WebSearch 채택/회피 신호를 주석 `라이브:`에 기록(웹 미가용 시 "미조회").

**2차 패스 — 셀프 크리틱.** 플랜을 브리프와 대조한다: *"비슷한 브리프 아무 데나 내도 통할 안인가?"*
그렇다면 그 부분(팔레트든 히어로든)을 수정하고 무엇을 왜 바꿨는지 한 줄 남긴다. 특히
AI 디폴트 룩 3종(크림+세리프+테라코타 / 근검정+애시드그린 / 신문식 hairline)과 겹치는지 검사.
통과한 플랜대로만 구현하고, 모든 색·타이포 결정은 플랜에서 도출한다.

## 1. 레이아웃·정렬

- 콘텐츠 폭은 섹션 의도에 따라 **가변** — 좁은 읽기 컬럼 / 기본(≈1180~1240px) / 넓게 / 풀블리드 중
  고른다. `.inner`(좌우 패딩 데스크톱 32 / 태블릿 24 / 모바일 20px)는 필요할 때 쓰는 가변 컨테이너이고,
  없는 섹션도 정상이다.
- 판단 기준은 '고정 레일'이 아니라 **의도된 폭 + 좌측 정렬 기준선**. 콘텐츠가 브라우저 끝에 붙는 것이
  의도라면 OK, 의도 없이 붙거나 넘치면 실패. 풀폭 배경은 `section`이, 폭 제한은 컨테이너가 담당.
- **페이지당 1~2개 섹션은 의도적으로 틀을 깬다** — 방향의 무브 세트(풀블리드·편측 흘림·오버랩 등)를
  사용한다. 틀을 깨는 섹션에서도 주요 텍스트의 좌측 기준선은 유지.
- 한 페이지의 주요 텍스트 시작점은 같은 좌측 기준선. 정보형 섹션은 좌측 정렬 기본, 가운데 정렬은
  필요한 곳만(D12 소프트 프로덕트는 중앙 정렬이 관례).
- 섹션 간격은 정보 흐름에 따라 72 / 96 / 120px(D2 콰이어트 럭셔리는 140px+)로 변주 — 전부 동일 간격 금지.
  본문 텍스트 폭 `max-width: 520~680px`.
- **메인 히어로는 첫 화면을 지배하는 것이 기본** — `min-height: 92~100vh`(에디토리얼·럭셔리 방향은
  76~88vh 허용). 이미지든 타이포든 방향의 히어로 문법을 따르고, "중앙 정렬 + pill 뱃지 + 큰 제목 +
  버튼 2개"의 템플릿 히어로는 금지. 서브·관리자·제품 유형에는 풀스크린을 강제하지 않는다.
- **관리자·도구형 제품**은 마케팅 문법 대신 [saas-admin.md](saas-admin.md)의 앱 셸(사이드바+상단바)과
  화면 골격을 따른다. 후기·CTA 같은 마케팅 섹션을 억지로 넣지 않는다.

## 2. 섹션 다양성 (AI스러움 방지)

- **고정 스켈레톤 금지**: "히어로 → 아이콘 3열 카드 → 로고월 → 통계 배너 → pricing → FAQ → CTA"는
  AI 서명이다. 섹션 구성은 업종·목적(인터뷰 답)에서 도출한다.
- 8개 섹션이면 레이아웃 패밀리 4종 이상: 텍스트 중심 / 좌텍스트·우이미지 / 넓은 이미지+짧은 설명 /
  리스트·단계 / 가로 스크롤 / 괘선 표 / 하단 CTA … zigzag(텍스트-이미지 교대)는 최대 2연속.
  **좌텍스트·우이미지(좌우 반전 편측 흘림 포함)는 페이지당 최대 2회 — 상한이지 금지가 아니다.**
  §0에서 선언한 섹션맵이 이 규칙을 만족해야 하며, 못 하면 섹션맵을 다시 짠다(중앙정렬 관례인 D12는 예외).
- 3열 카드 반복 금지. 섹션마다 밀도·여백·이미지 비율을 다르게 — 모든 섹션이 같은 max-width·radius·
  shadow·간격이면 실패.
- eyebrow(위 라벨)는 3섹션당 1개 이하. 의미 없는 01/02/03 번호 장식 금지(실제 절차·순서일 때만).
- 섹션 타입(참고): hero, intro, about, service, product, portfolio, gallery, process, pricing,
  review, faq, contact, cta, footer — 이 중 브리프에 필요한 것만.
- 차별화 장치는 방향의 무브 세트에서. 볼륨(절제/균형/과감)은 design-trends.md 사용법을 따른다.

## 3. 타이포

- 본문 기본은 Pretendard:
  ```html
  <link rel="stylesheet" href="https://cdn.jsdelivr.net/gh/orioncactus/pretendard@v1.3.9/dist/web/static/pretendard-dynamic-subset.min.css">
  ```
  `body { font-family: "Pretendard", -apple-system, BlinkMacSystemFont, system-ui, sans-serif; }`
- **헤드라인은 방향에서 도출한 디스플레이 폰트를 쓴다**(design-trends.md 타이포 팔레트의 검증 CDN).
  "Pretendard 단독"은 어드민 전용 — 사용자 페이지에서 헤드라인까지 Pretendard면 개성 실패.
- 제목 `font-weight`는 방향에 따라 극단을 겁내지 않는다(100~200의 얇음 또는 800~900의 두꺼움 —
  600 언저리의 애매함이 가장 AI스럽다). 본문 400~500 / `line-height: 1.6~1.75`.
- 국문 헤드라인은 `letter-spacing: -0.01~-0.03em` + `word-break: keep-all` + 줄바꿈 위치 직접 설계.
- 크기 단계는 3~4개로 압축, 단계 간 배율 1.3배 이상.

## 4. 폰트 세로 정렬 (중요 — 가장 흔한 결함)

- 뱃지·버튼·칩처럼 텍스트를 감싼 작은 박스는 **flex로 중앙 정렬**한다. padding만으로 맞추지 않는다:
  ```css
  .button, .badge, .chip {
    display: inline-flex;
    align-items: center;      /* 세로 정중앙 */
    justify-content: center;  /* 가로 정중앙 */
    line-height: 1;           /* 고정 높이 요소는 1로 두고 padding으로 높이 확보 */
  }
  ```
- 아이콘+텍스트는 `gap`으로 간격, 두 자식 모두 flex 중앙. 고정 `height` 버튼/뱃지는 `line-height`가
  글자를 밀지 않는지 자체 테스트에서 확인.

## 5. 색·디테일

- 팔레트는 1차 패스에서 선언한 4~6색만 쓴다. 액센트는 1색(D6 도파민 팝만 2~3색 충돌 허용).
- 구분은 그림자보다 여백·배경 톤·얇은 선·타이포 위계로 먼저. `border-radius`는 하나의 시스템으로
  통일(각짐 0~4 / 표준 6~10 / 둥긂 12~20 중 방향에 맞는 것 하나 — 페이지 안에서 섞지 않는다.
  pill은 인터랙티브 요소만).
- CSS 변수를 먼저 정의: `--page-max`, `--side-padding`, `--section-space`,
  `--color-primary/text/muted/line/bg/bg-soft` 등. 브랜드색 파생에 oklch를 쓸 때 폴백은
  **사용처에서**(`background: #hex; background: var(--brand-deep);`) — 커스텀 프로퍼티 선언
  자체에는 hex→oklch 덮어쓰기 폴백이 동작하지 않는다(design-trends.md 색 시스템 팁 참조).
- 공통 레이아웃 클래스: `.page`, `.section`, `.section--tone`, `.inner`(가변 폭·옵션), `.grid`,
  `.eyebrow`, `.section-title`, `.section-desc`, `.button`. 단, 모든 섹션을 같은 컴포넌트처럼 보이게
  하지 않는다.
- CSS 특이성 주의: `.section` 같은 타입 셀렉터와 요소 셀렉터가 서로 상쇄되지 않게 — 섹션 간
  padding/margin이 예고 없이 겹치는 흔한 원인.
- 모션은 design-trends.md D섹션의 안전 열화 규칙을 따른다: 최종 상태가 기본값,
  연출은 `@supports` + `prefers-reduced-motion` 게이트 안에만.

## 6. 이미지

- **무료 이미지 사이트의 실제 URL**로 넣는다(플레이스홀더 회색 박스 금지):
  - Unsplash: `https://images.unsplash.com/photo-<id>?w=1600&q=80` (알고 있는 실제 photo id 사용.
    `source.unsplash.com`은 서비스 종료 — 쓰지 않는다.)
  - Pexels: `https://images.pexels.com/photos/<id>/...jpeg?auto=compress&w=1600`
- **HTML에 넣기 전에 curl로 검증한다**: 후보 URL들을 `curl -s -o /dev/null -w "%{http_code}" <url>`로
  일괄 확인해 200이 아닌 것은 버린다. 유효한 이미지를 못 구한 영역은 회색 박스가 아니라 **방향의
  팔레트로 만든 듀오톤 컬러 블록 + 타이포 그래픽**으로 폴백한다(사진 주도 방향이면 유효 이미지를
  구할 때까지 키워드를 바꿔 재시도).
- 자체 테스트에서도 모든 이미지의 실제 로드(naturalWidth > 0)를 재확인하고, 깨진 것은 교체한다.
- hero / 소개 / 사례 영역은 비율을 서로 다르게. `figure`+`figcaption`, 설명형 `alt`.
- 이미지 톤은 인터뷰의 방향(실사/제품컷/일러스트)을 따르되, 방향의 질감 처리(duotone·그레인·
  마스크 셰이프)를 입혀 "스톡 사진 그대로" 느낌을 지운다.
- 코드로 어설프게 그린 손그림풍 SVG 일러스트는 역효과 — 일러스트가 필요하면 실제 이미지 에셋을
  쓰거나 도형·타이포 그래픽으로 대체한다.

## 7. 카피·태그

- 한국어. **제품·서비스가 실제로 하는 일을 구체적 동사+명사로** 쓴다. 사용자의 상황을 서술한다
  ("대기 없이 오늘 저녁 예약" > "스마트한 예약 솔루션").
- 금지 어휘·패턴: "혁신적인/최고의/새로운 기준/완벽한", 번역투("~을 통해", "당신의 비즈니스를"),
  em-dash(—) 남발, 모든 문장이 같은 종결어미로 끝나는 균일 리듬.
- 주어지지 않은 브랜드명·수치·후기·인증마크를 사실처럼 지어내지 않는다. 가짜 정밀 숫자
  (만족도 98%, 4.9/5.0) 금지 — 데이터가 주어졌을 때만 숫자를 쓴다.
- 버튼·CTA는 눌렀을 때 일어나는 일을 그대로 ("상담 예약하기", "메뉴 보기" — "자세히 보기" 남발 금지).
  같은 액션은 페이지 전체에서 같은 이름.
- 선택 태그는 방향 결정에만 쓰고 화면에 노출하지 않는다. 충돌하는 태그는 업종·목적에 더 맞는 쪽 우선.
  태그가 많으면 핵심 태그 3~5개를 중심축으로.

## 8. 업종별 정보 구조 (우선순위)

- 병원·교육·금융·B2B — 신뢰, 절차, 전문성, 검증 요소
- 숙박·여행·레저·식음료 — 이미지 몰입감, 가격/예약/위치, 경험 중심 카피
- 쇼핑·제품·브랜드 — 차별점, 사용 장면, 구성 옵션, 구매 판단 요소
- 공공·기관·협회 — 안정감, 명확한 안내, 접근성, 정보 전달력
- 스튜디오·IT·플랫폼 — 문제 해결 흐름, 기능 설명, 사용 사례, 운영 효율
- 로컬 매장·생활 서비스 — 위치, 운영 시간, 이용 절차, 문의 흐름 (+ 카카오톡 채널·전화 CTA 관례)
- 푸터에는 법정 정보 자리(상호·대표자·사업자등록번호·주소·개인정보처리방침)를 확보한다 —
  실데이터가 없으면 형태만 잡고 지어내지 않는다.

## 9. 출력 전 자체 점검

- 첫 화면만 보고 업종과 목적이 이해되는가? **그리고 이 시안만의 개성(시그니처)이 보이는가?**
- 선언한 방향·팔레트·타이포가 실제로 구현됐는가(플랜과 결과물의 일치)?
