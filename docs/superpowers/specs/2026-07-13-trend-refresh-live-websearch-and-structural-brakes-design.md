# 최신 트렌드 반영 + 라이브 WebSearch + 구조 브레이크 (설계)

- 날짜: 2026-07-13
- 대상 스킬: `cova-make-design`, `cova-make-guide`
- 상태: 설계 승인 대기

## 1. 배경 / 문제

시안 결과물에서 두 가지 반복 불편이 관찰됐다.

1. **마키(흐르는 띠 배너)가 거의 모든 시안에 등장한다.** 규칙상 마키는 D5/D6/D9 3개 방향의 무브
   세트에만 있고 "페이지당 1개"로 제한돼 있으나, 실제로는 전 방향에 새어 나온다.
2. **시안 대부분이 "좌 타이틀 / 우 이미지" 한 가지 레이아웃의 변주다.** 섹션마다 배치가 거의 같다.

원인 진단(규칙 파일 기준):

- **마키 과다** = ① 뷰어 CSP가 스크립트를 차단(SKILL.md)해 CSS-only 모션 중 가장 구현이 쉬운 게
  마키 → 볼륨 규칙의 "모션 연출" 할당량을 채우는 최저저항 답이 됨 ② 시그니처 1개의 만만한 정답 ③
  "다른 방향 무브 차용 허용" 문구로 홈 방향 밖으로 누출 ④ self-test 클리셰 스캔에 마키가 없어 통과.
- **레이아웃 반복** = ① 1차 패스 design-plan이 방향·팔레트·타이포·시그니처·섹션 순서까지만 커밋하고
  **섹션별 레이아웃 패밀리는 선언 대상이 아님** → 생성 순간 "안전한 평균값"으로 회귀 ② 강한 좌측
  기준선 규율이 텍스트-좌/이미지-우를 편안한 디폴트로 유도 ③ "zigzag 최대 2연속"은 3연속만 막을 뿐
  ④ self-test의 "4종 이상 변주"가 대조할 선언(플랜)이 없어 오카운트 가능.

추가로, 트렌드 레퍼런스(`design-trends.md`)의 연도 흐름·클리셰 목록이 갱신 시점 이후로 낡을 수 있다.

## 2. 목표 / 비목표

**목표**

- 트렌드 레퍼런스를 최신 리서치로 갱신하고, **부상 중인 신흥 방향을 카탈로그에 추가**한다.
- 생성 시점에 **에이전트 WebSearch로 라이브 트렌드를 조회**해 플랜에 녹인다(두 스킬 모두).
- 마키 과다와 레이아웃 반복을 **검증 가능한 구조 브레이크**로 근본 차단한다.

**비목표**

- COVA 백엔드/패턴 API 변경(트렌드 서빙 엔드포인트 신설 등)은 하지 않는다 — 이 레포 밖.
- 12방향 카탈로그를 무분별하게 확장하지 않는다(큐레이션 철학 유지, 순증 소폭).
- 기존 생성 파이프라인의 curl/독립 실행 원칙을 깨지 않는다(웹 없으면 문서로 폴백).

## 3. 접근 개요 (A안: 문서=지속 베이스라인 + WebSearch=신선도 레이어)

`design-trends.md`는 큐레이션된 뼈대(방향·무브·브레이크)로 남기고, ① 한 번의 리서치로 내용 갱신
② 생성 시 바운드된 WebSearch로 낡음을 교정 ③ 검증 가능한 구조 브레이크를 규칙·self-test에 심는다.

### 핵심 설계 원칙 — "라이브 트렌드 검색의 역설"

트렌드 아티클은 클리셰의 진원지다("2026 web design trends" 검색 상단 = AI 그라데이션·벤토·마키·
오버사이즈 타이포). 따라서 WebSearch는 **이중 용도**로만 쓴다:

- **채택 신호**: 지금 부상 중이거나 저무는 흐름 2~3개.
- **회피 신호**: 지금 포화·클리셰가 된 것 → 그대로 **회피 목록**에 넣는다.

이 원칙이 없으면 라이브 검색이 오히려 마키·보라 그라데이션을 재주입한다.

## 4. 상세 설계

### 4.1 문서 갱신 — `design-trends.md` (make-design·make-guide 두 copy, 현재 완전 동일 473줄)

두 파일은 **동일하게** 갱신해 동기화를 유지한다(구현 시 한쪽 작성 후 복사·diff 확인).

갱신 대상 절:

- **`2026 감각 잡기` 연도 흐름** — 2026 하반기(작성 시점) 관점으로 갱신하고 그 시점의 신호 추가.
- **클리셰 회피 목록**(`국내 문법`의 "버려도 되는 관행" + `출력 전 자체 점검`의 클리셰 스캔) —
  마키·보라 그라데이션·벤토를 "표준/포화"로 재분류하고, 리서치가 찾은 **새 클리셰를 추가**.
- **12방향 카탈로그** — 저문 방향은 톤다운·주의 강화. 리서치가 식별한 **신흥 방향 1~3종을 신규 추가**
  (기존 12종과 실제로 구별되는 것만). 각 신규 방향은 기존과 동일한 레시피 형식을 갖춘다:
  무드 / 팔레트 공식(이름+hex 4~6) / 타이포(검증 CDN 폰트) / 레이아웃 문법 / 질감·장식 /
  무브 세트(카탈로그 번호) / 어울리는 업종 / 주의. 최종 규모는 12 → 약 13~15종(과확장 금지).
- **무브 14(마키) 재작성** — §4.3(a).
- **신설 절 "라이브 트렌드 접합"** — WebSearch 결과를 게이트·플랜에 녹이고 회피 목록으로 쓰는 절차.
  (`사용법 — 생성 게이트` 근처에 배치.)

### 4.2 라이브 트렌드 스캔 단계 (두 스킬 모두)

- **make-design**: 3단계(패턴 검색)와 4단계(HTML 생성) 사이 **"3.5 라이브 트렌드 스캔"** 신설.
- **make-guide**: 3단계(COVA 패턴 참고)와 4단계(디자인 시스템 확정) 사이 동일 단계 신설.
- **적용 범위**: 방향 카탈로그를 쓰는 **user·소비자형 제품** 스코프에서 수행. **admin·도구형 제품**은
  `saas-admin.md`가 지배하므로 라이브 스캔을 생략하거나 "admin/dashboard 2026" 류로 가볍게만 한다.
- **동작**:
  1. 인터뷰로 확정된 업종·방향 후보로 타깃 검색 1~2회
     (예: `<업종> 웹디자인 2026`, `<방향후보> web design award 2026`).
  2. 결과에서 **채택 신호 2~3개 + 회피 신호(포화된 것) 별도** 추출(§3 이중 용도).
  3. 게이트의 "방향 후보 3개 나열·기각"과 시그니처·무브 선택의 근거로 사용.
- **기록**: 산출물 메타에 라이브 필드 남김.
  - 시안: HTML 최상단 design-plan 주석에 `라이브: 채택[…] / 회피[…]`.
  - 가이드: `DESIGN.md` frontmatter에 동등 필드.
- **폴백**: 웹 도구 없음/결과 빈약 → 문서 기준 진행, 메타에 `라이브: 미조회(문서 기준)` 표기.
- **바운드**: 검색 상한 2~3회(비용). 검색에서 본 **수치·브랜드명·후기를 시안 카피에 사실처럼
  넣는 것은 금지**(design-rules.md §7 유지) — 라이브 신호는 방향·연출 결정에만 쓴다.

### 4.3 구조 브레이크 3종

**(a) 마키 억제**

- `design-trends.md` 무브 14 재작성: "**D5/D6/D9 홈 방향에서만** 사용, 그 외 차용 금지. 페이지당 1개,
  읽어야 하는 정보 금지."
- 볼륨 규칙(`사용법 — 생성 게이트` 3항, 절제/균형/과감의 모션 할당량) 옆에 명시:
  "JS 차단 환경의 CSS-only 모션은 **스크롤 리빌(19)·오로라(20)·그라데이션 슬라이드(21, 페이지당 1)·
  호버 스포트라이트(28)를 우선**, 마키는 홈 방향(D5/6/9)에 한정." → 모션 할당량의 디폴트 답을 마키에서 뗌.
- `self-test.md` §2 클리셰 스캔 항목 추가: "**마키 2개 이상, 또는 홈 방향 밖 마키 = 실패.**
  `@keyframes marquee` / `translateX(-50%)` 카운트로 확인."

**(b) 섹션 레이아웃 다양성 강제**

- `design-rules.md` §0 1차 패스 design-plan 주석에 **`섹션맵:` 필드 신설** — 섹션별 레이아웃 패밀리를
  코드 전에 선언한다. 예: `섹션맵: hero / 좌텍우이미지 / 풀블리드미디어 / 리스트단계 / 괘선표 / 하단CTA`.
- 하드캡을 §2에 추가: "**좌텍스트·우이미지(좌우 반전 편측 흘림 포함)는 페이지당 최대 2회.**
  8섹션이면 서로 다른 패밀리를 4종 이상 실제로 선언한다." (기존 산문 지침을 검증 가능한 선언으로 승격.)
- `self-test.md`가 주석의 `섹션맵`과 실제 DOM을 대조: "**좌우만 반전한 같은 패밀리는 1종으로 카운트.
  4종 미만 = 실패.**"

**(c) 주석/frontmatter 스키마 연동**

- 시안 design-plan 주석에 `섹션맵:`·`라이브:` 두 필드 추가.
- `self-test.md` §2가 두 필드의 **존재와 실제 구현 일치**를 확인(플랜 따로 결과 따로 금지).
- 가이드는 섹션맵이 무의미(페이지 생성 아님)하므로 `라이브:` 필드만 `DESIGN.md` frontmatter에 추가.

## 7. 폴백 · 비용 · 리스크

- **웹 미가용**: 라이브 스캔은 조용히 문서 기준으로 폴백, 메타에 `미조회` 표기. 실패로 취급하지 않음.
- **비용**: 생성당 검색 2~3회 상한. 인터뷰 뒤 1회 배치로 수행해 왕복 최소화.
- **역설 리스크**: 라이브 검색이 클리셰를 재주입 → §3 이중 용도(회피 신호 강제 추출)로 차단.
- **동기화 드리프트**: 두 `design-trends.md`가 어긋남 → 구현 시 한쪽 작성 후 복사, diff로 동일 확인.
- **과제약 리스크**: 섹션맵 하드캡이 특정 브리프에서 과할 수 있음 → "최대 2회"는 상한이지 금지가 아님을
  명시하고, 방향 문법이 중앙정렬을 요구하는 D12 등은 예외 문구 유지.
